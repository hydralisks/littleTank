# Little Tank MVP 设计文档

## 1. 项目定位

`Little Tank` 是一个受 `Little Healer` 启发的单人电脑端小游戏原型。

玩家不扮演治疗，而扮演 MMO 小队中的 `主坦克（Main Tank）`。游戏的核心不是角色移动或场景演出，而是围绕一组 `以团队框体形式堆叠显示的敌方血条` 进行战场秩序管理。

玩家通过：

- 切换目标
- 制造和维持仇恨
- 嘲讽失控敌人
- 打断关键读条
- 施加控制链
- 开启减伤
- 抓住窗口打爆发

来确保敌人被稳定“接住”，避免其威胁队友，并最终完成击杀。

## 2. 核心体验目标

MVP 需要优先验证以下体验是否成立：

- 玩家是否能在纯信息 UI 上感受到“主坦职责”
- 玩家是否能在多目标压力下形成明确的优先级判断
- 玩家是否能从 `仇恨管理 + 打断 + 控制 + 生存` 中获得操作乐趣
- 玩家失败时是否能清楚知道自己失误在哪里

本项目的第一目标不是做完整 MMO 战斗，而是做一个清晰、可玩、可迭代的 `坦克职责模拟器`。

## 3. MVP 范围

### 3.1 包含内容

- 单场战斗原型，时长约 `3-5 分钟`
- `5 个敌方框体单位`
- 其中 `1 个骷髅头单位` 作为高危精英/首领职责承担者，但仍然属于普通敌方列表中的一个单位
- 一套基础仇恨系统
- 一套基础读条/打断系统
- 一套基础控制系统
- 一套基础玩家生存系统
- 一套基础评分系统
- 底部横向技能栏，技能显示剩余 CD

### 3.2 不包含内容

- 角色移动
- 场景地图探索
- 装备系统
- 天赋树
- 多职业队友细化 AI
- 复杂剧情
- 独立 Boss 场景或 Boss 专属界面
- 完整数值成长

## 4. 核心玩法循环

玩家在每个时刻都在执行以下循环：

1. 观察敌方团队框体，找出当前最危险目标
2. 判断危险来源：
   - 仇恨丢失
   - 高危读条
   - 敌人锁定队友
   - 高伤害窗口即将到来
3. 向下扫视玩家状态区，确认自身：
   - 当前血量
   - 资源是否足够
   - 减伤是否可用
   - GCD 是否结束
4. 再向下看技能栏，确认：
   - 嘲讽是否转好
   - 打断是否转好
   - 控制是否可用
   - 爆发技能是否值得交
5. 执行操作
6. 观察反馈并调整下一次决策

这个循环要形成一种类似 `Little Healer` 的高频信息处理体验，但主题从“看队友血条”切换为“看敌方威胁框体”。

## 5. 界面设计

## 5.1 总体布局

界面自上而下分为三层：

1. `敌方团队框体区`
2. `玩家状态区`
3. `横向技能栏`

不在顶部显示玩家血量、资源等核心状态，避免玩家视线在上下两端来回跳跃。

## 5.2 敌方团队框体区

这是整个游戏最重要的视觉区块。

每个敌方框体建议显示：

- 名称
- 怪物类型图标
- 当前 HP / Max HP
- 血条
- 当前目标：`TANK / ALLY`
- 仇恨状态色
- 当前读条
- 状态图标：嘲讽、昏迷、沉默、护盾、狂暴等
- 高危标记
- 骷髅头标记（用于高危单位）

### 颜色建议

- 绿色：当前稳定攻击玩家
- 黄色：玩家仇恨接近被队友超过
- 红色：当前正在攻击队友
- 紫色边框：正在读高危技能
- 蓝灰色覆盖：处于控制中

### 特殊单位

“Boss” 不单独做面板。

高危单位只是在敌方框体中表现为：

- 左侧骷髅头标记
- 更长血条
- 更危险技能列表
- 更高伤害和机制权重

这样可以保持所有战斗信息都收束在同一个框体系统里。

## 5.3 玩家状态区

玩家状态区位于 `敌方团队框体` 和 `技能栏` 之间。

建议显示：

- 玩家 HP
- 主资源（怒气/能量）
- 当前减伤 Buff
- 当前目标名
- GCD 进度
- 队伍压力值

其中 `队伍压力值` 是一个重要抽象指标，用于替代复杂队友框架。

### 队伍压力值含义

队伍压力值会因为以下情况升高：

- 有敌人没有攻击玩家而在攻击队友
- 高危读条未被打断
- 高危敌人进入狂暴/强化状态
- 队伍持续受到范围伤害

当队伍压力值过高时：

- 屏幕出现警报色
- 玩家评分下降
- 达到阈值后直接失败，视为队伍崩盘

## 5.4 横向技能栏

技能栏固定在底部，横向排布。

每个技能格显示：

- 图标
- 快捷键
- 技能名缩写
- 剩余 CD 数字
- 资源消耗
- 不可用状态遮罩

建议 MVP 先做 `8 格技能位`：

- `1` 嘲讽
- `2` 打断
- `3` 单体控制
- `4` 群体拉怪
- `Q` 减伤
- `E` 顺劈
- `R` 爆发
- `F` 应急技能

## 6. 玩法系统设计

## 6.1 仇恨系统

仇恨是游戏的第一核心系统。

每个敌人维护一个简化仇恨表：

```ts
type ThreatTable = {
  tank: number;
  dps: number;
  healer: number;
};
```

### 规则

- 玩家使用技能会提高 `tank` 仇恨
- 队友会自动持续制造 `dps` 或 `healer` 仇恨
- 敌人默认攻击当前仇恨最高目标
- `嘲讽` 会在短时间内强制敌人攻击玩家，并抬升玩家仇恨
- 群体拉怪技能会同时抬高多个敌人的玩家仇恨

### 仇恨表现

- 如果玩家是最高仇恨，则敌方框体显示绿色
- 如果队友仇恨接近追平，则框体转黄
- 如果敌人已转火队友，则框体变红并提高队伍压力

## 6.2 读条系统

读条是游戏的第二核心压力来源。

敌人会周期性释放技能：

```ts
type Cast = {
  spellId: string;
  spellName: string;
  remainingMs: number;
  totalMs: number;
  interruptible: boolean;
  dangerLevel: "low" | "medium" | "high";
};
```

### 技能类型

- 对队友的高伤害法术
- 对玩家的坦克重击
- 敌方自疗
- 敌方增益/狂暴
- 召唤额外单位

### 玩家应对

- 可打断技能：优先用打断
- 不可打断但可控技能：用控制打断动作
- 坦克重击：提前开减伤
- 自疗技能：及时转火或打断

## 6.3 控制系统

MVP 建议只做三种控制：

- `stun`：无法行动
- `silence`：无法施法
- `taunt`：强制攻击玩家

敌人需有基础控制免疫标签：

```ts
type ControlProfile = {
  stunImmune: boolean;
  silenceImmune: boolean;
  tauntImmune: boolean;
};
```

高危骷髅头单位建议：

- 不完全免疫嘲讽
- 可被减速或压制式短控影响
- 对硬控有抗性或时长缩短

## 6.4 玩家生存系统

玩家是主坦，因此自身生存必须成立。

### 玩家需要处理的伤害来源

- 平 A 压力
- 骷髅头单位的坦克重击
- 没有及时处理的施法伤害
- 多个小怪同时在身上的群体压力

### 失败条件

满足任一条件即失败：

- 玩家死亡
- 队伍压力值到达上限
- 关键事件未处理导致崩盘

## 6.5 队友抽象系统

MVP 不建议为每个队友制作完整血条和技能 AI。

建议使用抽象小队模型：

```ts
type PartyState = {
  pressure: number;
  damagePerSecond: number;
  healingPerSecond: number;
};
```

其作用是：

- 自动对敌人输出
- 自动产生仇恨
- 通过 `pressure` 反映是否被怪物压垮

后续如要升级，可再扩展为 3-4 个具体队友单位。

## 7. 敌人设计

MVP 先做 5 类敌人模板。

## 7.1 普通近战怪 `grunt`

- 血量中低
- 技能少
- 主要用于教学仇恨维持

## 7.2 施法者 `caster`

- 血量较低
- 会读高危法术
- 需要玩家及时打断

## 7.3 刺客 `assassin`

- 容易切向队友
- 会触发短时间高压事件
- 需要嘲讽或单体控制处理

## 7.4 暴徒 `brute`

- 输出高
- 攻击间隔慢但单次伤害重
- 容易与玩家生存系统形成压力联动

## 7.5 骷髅头单位 `skull`

- 作为高危精英/首领职责承担者
- 仍属于敌方框体中的一个普通项
- 有更高 HP
- 有坦克重击
- 有阶段性高危读条
- 权重最高

## 7.6 推荐 MVP 敌方阵容

一场战斗建议固定为：

- `1 x skull`
- `1 x caster`
- `1 x assassin`
- `2 x grunt/brute`

这样足够形成多线程压力：

- 骷髅头单位负责压主坦
- 施法者负责制造打断需求
- 刺客负责制造失控需求
- 其余单位提供基础仇恨维护压力

## 8. 技能设计

## 8.1 技能列表

### 嘲讽

- 快捷键：`1`
- 目标：当前目标
- 效果：强制目标攻击玩家 `3s`
- 附带：立即提升玩家仇恨
- 作用：处理转火

### 打断

- 快捷键：`2`
- 目标：当前目标
- 效果：打断可打断读条，并使其短暂锁法
- 作用：处理高危施法

### 单体控制

- 快捷键：`3`
- 目标：当前目标
- 效果：昏迷/压制 `2s`
- 作用：争取处理窗口

### 群体拉怪

- 快捷键：`4`
- 目标：全部敌人
- 效果：短时间强制多个敌人攻击玩家
- 作用：处理多目标失控

### 减伤

- 快捷键：`Q`
- 目标：自己
- 效果：减少受到伤害 `X%`
- 作用：应对坦克重击和高压时间段

### 顺劈

- 快捷键：`E`
- 目标：当前目标及相邻目标
- 效果：造成伤害并制造范围仇恨
- 作用：维持群怪稳定

### 爆发

- 快捷键：`R`
- 目标：当前目标
- 效果：短时间造成高伤害和高仇恨
- 作用：压低高危单位血量或抢回节奏

### 应急技能

- 快捷键：`F`
- 目标：自己或当前目标
- 效果：临时保命/解危
- 作用：容错

## 8.2 技能数据结构

```ts
type SkillId =
  | "taunt"
  | "interrupt"
  | "stun"
  | "massTaunt"
  | "shieldWall"
  | "cleave"
  | "burst"
  | "panic";

type Skill = {
  id: SkillId;
  name: string;
  hotkey: string;
  cooldownMs: number;
  remainingCooldownMs: number;
  resourceCost: number;
  targetType: "self" | "currentEnemy" | "allEnemies";
};
```

## 9. 一场 3-5 分钟战斗的流程

## 9.1 开场 0-30 秒

目的：教玩家认 UI 和切目标。

- 所有敌人入场
- 只有普通近战怪先开始造成稳定压力
- 玩家学习点击/Tab 切目标
- 玩家学习用顺劈和基础攻击维持仇恨

## 9.2 第一阶段 30-60 秒

目的：引入打断。

- 施法者开始周期性读条
- 玩家如果不打断，队伍压力明显上升
- 玩家学会锁定施法者并用打断处理

## 9.3 第二阶段 60-100 秒

目的：引入转火失控。

- 刺客会短暂转火队友
- 玩家必须迅速切目标并使用嘲讽或控制
- 如果处理慢，队伍压力暴涨

## 9.4 第三阶段 100-160 秒

目的：引入坦克生存。

- 骷髅头单位开始固定周期释放 `坦克重击`
- 玩家必须在重击前开减伤
- 如果减伤交错，玩家会被快速压低血量

## 9.5 结束阶段 160 秒后

目的：形成综合考试。

- 施法者、刺客、骷髅头单位压力叠加
- 玩家需在多目标中做优先级判断
- 击杀顺序与技能保留将显著影响能否通关

## 10. 胜负与评分

## 10.1 胜利条件

- 所有敌人死亡

## 10.2 失败条件

- 玩家 HP 归零
- 队伍压力值达到上限

## 10.3 评分维度

- 仇恨失控次数
- 成功打断次数
- 漏断高危技能次数
- 正确减伤次数
- 队伍压力峰值
- 通关时间

建议结算时给一个 `坦克评级`：

- `S` 战场支柱
- `A` 稳定接敌
- `B` 勉强维稳
- `C` 防线摇摆
- `D` 队伍崩盘

## 11. 数据结构草案

```ts
type EnemyKind = "grunt" | "caster" | "assassin" | "brute" | "skull";

type StatusEffect =
  | { id: "taunted"; remainingMs: number }
  | { id: "stunned"; remainingMs: number }
  | { id: "silenced"; remainingMs: number }
  | { id: "shieldWall"; remainingMs: number }
  | { id: "enraged"; remainingMs: number };

type Enemy = {
  id: string;
  kind: EnemyKind;
  name: string;
  hp: number;
  maxHp: number;
  target: "tank" | "dps" | "healer";
  threat: {
    tank: number;
    dps: number;
    healer: number;
  };
  cast: Cast | null;
  statuses: StatusEffect[];
  controlProfile: ControlProfile;
};

type Player = {
  hp: number;
  maxHp: number;
  resource: number;
  maxResource: number;
  currentTargetId: string | null;
  gcdRemainingMs: number;
  statuses: StatusEffect[];
  skills: Skill[];
};

type EncounterState = {
  timeMs: number;
  player: Player;
  enemies: Enemy[];
  party: PartyState;
  combatLog: string[];
  result: "playing" | "won" | "lost";
};
```

## 12. 前端实现建议

MVP 推荐使用：

- `TypeScript`
- `Vite`
- `React`

原因：

- 本项目主体是信息 UI 和状态机，不依赖复杂场景渲染
- DOM 对框体 UI、倒计时、状态更新更高效
- 快速验证玩法，后续再决定是否迁移至 Phaser 或 Godot

## 13. 模块拆分建议

## 13.1 页面模块

### `App`

- 管理页面框架
- 初始化战斗
- 组合三大 UI 区块

### `EncounterScreen`

- 承载整场战斗
- 订阅战斗状态
- 负责全局输入和结算切换

## 13.2 UI 模块

### `EnemyRaidFrameList`

- 渲染敌方框体列表
- 处理目标选中
- 展示威胁状态、读条、标记、状态图标

### `EnemyRaidFrameItem`

- 单个敌人框体
- 独立渲染血条、目标、读条条、图标层

### `PlayerStatusPanel`

- 渲染玩家 HP、资源、减伤、GCD、当前目标、队伍压力

### `SkillBar`

- 渲染底部横向技能栏
- 显示剩余 CD、可用状态、热键提示

### `CombatLogPanel`

- 可选
- 记录关键事件，便于调试和回放

## 13.3 逻辑模块

### `encounterEngine`

- 每 tick 更新战斗状态
- 推进敌人 AI、读条、攻击、仇恨变化、胜负判断

### `enemyAi`

- 决定敌人何时攻击、读条、切换行为

### `threatSystem`

- 处理仇恨生成、比较和目标切换

### `skillSystem`

- 执行玩家技能逻辑
- 处理 CD、资源、GCD、命中结果

### `statusSystem`

- 处理 Buff/Debuff 计时和效果

### `scoringSystem`

- 记录本场评分数据

## 15. 开发顺序建议

按以下顺序做，能最快得到可玩的版本：

1. 搭建三段式基础 UI
2. 完成敌方框体渲染和当前目标选中
3. 完成技能栏展示和 CD 倒计时
4. 完成玩家状态区展示
5. 完成战斗状态数据结构

## 16. 验收标准

MVP 达到以下标准即可认为完成第一阶段：

- 玩家可以点击或快捷切换敌方目标
- 技能栏能正确显示剩余 CD
- 敌方框体能正确表现仇恨状态和读条状态
- 至少存在一个带骷髅头标记的高危单位
- 玩家可通过嘲讽、打断、控制、减伤完成战斗
- 失败时能明确看出是 `自己倒下` 还是 `队伍崩盘`
- 通关后能输出基础评级
