# 关卡设计入口

## 2026-05-08 追加提醒：地图区域归属

- 当前 demo 选关地图第一页固定为 3 个区域，每个区域固定 6 个关卡。
- `stage_content.xlsx -> 区域` 负责区域表现信息：区域名、短名、地图标签、说明和强调色。
- `stage_content.xlsx -> 关卡.areaId` 必填，引用 `区域.areaId`，决定该 `stageId` 属于哪个区域。
- `stage_content.xlsx -> 关卡.order` 必填，表示该区域内第几关，当前只能填 `1~6`。
- 每个区域必须恰好填写 `order=1..6` 六个关卡；同一区域内不能重复 order。
- UI 节点位置按 `areaId + order` 取固定布局，例如 `areaId=midland, order=4` 会显示在第二个区域第 4 关位置。
- 本次只定向更新了 `public/designer-data/stage_content.xlsx`，没有运行 `npm run generate:designer-data` 或其他 public 生成命令。

这份文档给关卡策划、数值策划和后续接手 agent 使用，说明关卡相关内容应该在哪些 `.xlsx` 工作表里填写，以及哪些字段已经被当前 demo 运行时读取。

## 2026-04-24 追加提醒：队伍压力填写规则

- `encounter_balance.xlsx -> 关卡开场.partyPressure` 只表示开场初始压力，不表示“每秒自动涨压速度”。
- 当前 demo 默认不会因为时间流逝自动增加队伍压力。
- 如果关卡希望战斗过程中持续涨压或减压，应优先通过以下入口实现：
  - `开场状态`
  - `关卡词缀绑定 / 词缀定义`
  - `特殊规则绑定 / 特殊规则定义`
- 如果只是某个敌方技能打到玩家或打漏到队伍时要改压力，继续填写敌方技能表里的 `pressureOnHit / pressureOnMiss`。

## 1. 总入口

关卡设计主要编辑：

- `public/designer-data/stage_content.xlsx`
- `public/designer-data/encounter_balance.xlsx`

其中：

- `stage_content.xlsx` 负责地图、区域、关卡标题、展示文案和图例文案。
- `encounter_balance.xlsx` 负责每关开场数值、敌人布置、开场状态、关卡词缀、特殊规则和队伍自动伤害参数。

修改表后，常规验证只运行：

```powershell
npm run build
```

如果策划是直接编辑 `public/designer-data/*.xlsx`，不要重新生成样例表。只有在用户明确要求“重建/生成 public 下策划表样例”时，后续 agent 才能运行 `npm run generate:designer-data` 或 `node scripts/generateDesignerWorkbooks.mjs`，否则可能覆盖策划已有工作内容。

| --- | --- | --- | --- |

| `title` | 区域标题 | 是 | 地图区域完整名称 |
| `shortTitle` | 区域短名 | 是 | 地图节点或摘要使用 |
| `mapLabel` | 地图标注 | 是 | 地图上的地理名 |
| `description` | 区域说明 | 是 | 选关页右侧说明 |
| `accent` | 区域强调色 | 是 | UI 用颜色 |

### `关卡`

| 字段名 | 中文含义 | 当前是否接入 | 说明 |
| --- | --- | --- | --- |
| `stageId` | 关卡 ID | 是 | 需要和 `encounter_balance.xlsx` 中的 `stageId` 对应 |
| `areaId` | 所属区域 ID | 是 | 引用 `区域.areaId`；当前 demo 支持 `harbor / midland / highland` |
| `order` | 区域内序号 | 是 | `1~6`，每个区域固定六关且不能重复 |
| `title` | 关卡标题 | 是 | 例如 `潮湾堤口` |
| `subtitle` | 关卡副标题 | 是 | 用于简述训练重点 |
| `affix1Title` / `affix2Title` | 展示用词缀标题 | 是 | 选关页与状态页展示文案 |
| `affix1Description` / `affix2Description` | 展示用词缀说明 | 是 | 仅展示，不直接决定运行逻辑 |
| `affix1IconId` / `affix2IconId` | 展示用词缀图标 | 是 | 引用图标 ID |
| `rule1Title` / `rule2Title` | 展示用规则标题 | 是 | 当前只是展示文案 |
| `rule1Description` / `rule2Description` | 展示用规则说明 | 是 | 当前只是展示文案 |
| `rule1IconId` / `rule2IconId` | 展示用规则图标 | 是 | 引用图标 ID |
| `unlockedActiveSkillIdsCsv` | 本关新增解锁主动技能 | 是 | 逗号分隔，引用 `player_build.xlsx -> 主动技能定义.skillId` |

主动技能解锁规则：

- `unlockedActiveSkillIdsCsv` 只填写“这一关新增开放”的主动技能，不需要重复填写前面关卡已经开放的技能。
- 进入某关时，程序会按 `stageOrder` 从第一关累计到当前关，合并沿途所有 `unlockedActiveSkillIdsCsv`。
- 例如进入 `2-4` 时，可用主动技能 = 第一大关 `1-1` 到 `1-6` 解锁的所有技能 + `2-1 / 2-2 / 2-3 / 2-4` 新解锁技能。
- 该限制只看当前关卡前缀，不看玩家最高进度；即使已完成 `2-6`，重玩 `1-6` 时仍只能使用 `1-1` 到 `1-6` 前缀累计解锁的主动技能。

被动天赋解锁规则：

- 被动天赋不在本表逐个填写解锁 ID；可用性来自 `player_build.xlsx -> 被动天赋定义.tier` 与当前关卡位置。
- 初始前三关不开放 `tier=0` 被动。
- 完成 `RingingDeeps-3` 后开放 `tier=0`，也就是 `stage_content.xlsx -> 关卡` 按区域顺序和 `order` 排序后的第 4 关开始可使用 `tier=0`。
- 各区域第 6 关作为 boss 前节点，依次开放更高 tier：第一大区第 6 关开放到 `tier=1`，第二大区第 6 关开放到 `tier=2`，后续类推。
- 该限制只看当前进入关卡，不看玩家历史最高进度。

## 3. `encounter_balance.xlsx`

### `关卡开场`

这张表决定一关开始时的玩家、队伍和构筑规则。状态或持续效果类字段中，`durationMs=-1` / `durationMsOverride=-1` 表示永久持续到战斗结束，除非被明确指向该状态的效果移除。

| 字段名 | 中文含义 | 当前是否接入 | 说明 |
| --- | --- | --- | --- |
| `stageId` | 关卡 ID | 是 | 主键，需要和 `stage_content.xlsx -> 关卡.stageId` 对应 |
| `playerHp` | 玩家开场生命值 | 是 | 当前直接作为开场 HP |
| `playerMaxHp` | 玩家生命上限 | 是 | 战斗开始时作为玩家最大生命值基底，被动天赋在此基础上修正；比例上限被动会同步等比缩放 `playerHp` |
| `playerResource` | 玩家开场资源 | 是 | 当前用于怒气等资源 |
| `playerGcdRemainingMs` | 玩家开场 GCD | 是 | 单位毫秒 |
| `partyHp` | 队伍开场生命值 | 是 | 队伍血量 |
| `partyMaxHp` | 队伍生命上限 | 是 | 战斗开始时作为队伍最大生命值基底，被动天赋在此基础上修正 |
| `partyPressure` | 队伍开场压力 | 是 | 队伍压力 |
| `partyMaxPressure` | 队伍压力上限 | 是 | 战斗开始时作为队伍压力上限基底，被动天赋在此基础上修正；比例上限被动会同步等比缩放 `partyPressure` |
| `buildRuleId` | 本关构筑规则 ID | 是 | 引用 `player_build.xlsx -> 构筑规则定义.buildRuleId` |
| `partyAutoDamageIntervalMs` | 队伍自动伤害间隔 | 是 | 单位毫秒，例如 `1000` |
| `partyAutoDamageTargetCount` | 队伍自动伤害目标数 | 是 | 每次随机命中的存活敌人数 |
| `partyAutoDamageMin` | 队伍自动伤害最小值 | 是 | 每次命中的随机下限 |
| `partyAutoDamageMax` | 队伍自动伤害最大值 | 是 | 每次命中的随机上限 |
| `playerAutoDamage` | 玩家每秒自动攻击伤害 | 否 | 默认 `3` |
| `playerAutoHeal` | 玩家每秒自动受到治疗 | 否 | 默认 `0` |
| `partyAutoHeal` | 队伍每秒自动受到治疗 | 否 | 默认 `0` |

当前队伍自动伤害会对存活敌人造成随机伤害，并按 `threat = damage × 1 + 0` 产生队友仇恨。

### `敌人布置`

这张表决定每关刷哪些敌人、放在 5x5 血条格子的哪里，以及是否覆盖敌人目录里的默认值。

| 字段名 | 中文含义 | 当前是否接入 | 说明 |
| --- | --- | --- | --- |
| `stageId` | 关卡 ID | 是 | 所属关卡 |
| `spawnId` | 敌人实例 ID | 是 | 关卡内唯一，UI 不显示额外编号但逻辑保留 |
| `enemyId` | 敌人目录 ID | 是 | 引用 `enemy_data.xlsx -> 敌人定义.enemyId` |
| `row` | 行 | 是 | 1 到 5 |
| `col` | 列 | 是 | 1 到 5 |
| `nameOverride` | 名称覆盖 | 是 | 留空使用敌人定义名称 |
| `hpOverride` | 当前生命值覆盖 | 是 | 留空使用敌人定义血量 |
| `maxHpOverride` | 最大生命值覆盖 | 是 | 留空使用敌人定义血量 |
| `openingCastSkillNum` | 开场技能循环序号 | 是 | 1 基序号；`1` 表示 `enemy_data.xlsx -> 敌人定义.skillCycleCsv` 的第一个技能 |
| `openingRecoveryRemainingMs` | 开场发呆时间 | 是 | 单位毫秒；结束后才开始按 `openingCastSkillNum` 从头完整读条 |
| `openingTankThreat` | 开场玩家仇恨 | 是 | 留空按 0 |
| `openingAllyThreat` | 开场队伍仇恨 | 是 | 留空按 0 |

`target / threatState / openingCastSkillId / openingCastRemainingMs` 已移除。开场目标由 `openingTankThreat / openingAllyThreat`、`threatLogic` 自动计算；重复技能 ID 的循环用 `openingCastSkillNum` 消除歧义。
`skillCycleCsv / threatLogic / isSkullOverride` 也已从本表移除。技能循环、仇恨类型和精英/首领标记统一在 `enemy_data.xlsx -> 敌人定义` 维护。

| `ruleLogicId` | 当前样例行为 | 说明 |
| --- | --- | --- |

## 4. 填表顺序建议

1. 先在 `stage_content.xlsx -> 关卡` 写好关卡名、副标题、展示文案和本关新增的 `unlockedActiveSkillIdsCsv`。
2. 在 `encounter_balance.xlsx -> 关卡开场` 写开场玩家/队伍数值、`buildRuleId` 和队伍自动伤害参数。
3. 在 `敌人布置` 放置敌人实例，确认 `row / col` 不超出 1 到 5。
4. 需要开场已有效果时再填 `开场状态`。
5. 需要固定延时状态时填 `词缀定义`，再在 `关卡词缀绑定` 绑定到关卡。
6. 需要程序特殊逻辑时填 `特殊规则定义`，再在 `特殊规则绑定` 绑定到关卡。

## 5. 当前注意事项

- `ruleLogicId` 和 `effectLogicId` 都是程序逻辑入口 ID，不是脚本内容。
- 当前没有单独 boss 机制；`enemy_data.xlsx -> 敌人定义.isSkull` 作为精英/首领标记参与显示和部分技能限制。
- `stage_content.xlsx` 中的环境文案目前只是展示，不是独立运行系统。
- 如果表内字段和文档冲突，以 `src/game/data/workbookLoader.ts` 与 `scripts/generateDesignerWorkbooks.mjs` 为准。
