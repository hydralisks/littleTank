# 新坦克职业扩展接手入口

更新时间：2026-07-22

本文给下一位 agent 使用，目标是基于现有 `warrior_t` 战士T框架继续扩展新坦克职业，优先候选为熊T和死亡骑士T。本文记录设计路线、接手边界及已获授权的挑战表快照同步；`player_build.xlsx` 尚未写入新职业内容。

## 当前结论

建议开发顺序：

1. 先做熊T。
2. 再做死亡骑士T。

理由：

- 当前项目已有怒气资源、主动减伤、吸收、自疗、控制、打断、范围伤害、队伍保护、战斗统计和自动评测基础。熊T可以复用大部分战士T的现有运行时能力，只需要补职业数据、少量技能逻辑和若干被动逻辑。
- 死亡骑士T的核心差异是反应式自疗、近期承伤窗口、骨盾层数和更复杂资源节奏。它更适合作为第二个新职业，用来推动资源系统和职业 AI 策略升级。

## 已定稿基线：职业扩展架构第一部分

### 职业 ID 与状态复用

- 战士T：`warrior_t`。
- 熊T：`druid_bear_t`，职业专属技能、天赋和状态统一使用 `druid_bear_t_*` 前缀。
- 死亡骑士T：`dk_blood_t`，职业专属技能、天赋和状态统一使用 `dk_blood_t_*` 前缀。
- 语义完全一致的公共状态只保留一个规范定义，不附加职业前缀，例如 `taunted`、`countered`、`stunned`。
- 铁鬃、骨盾等职业专属状态必须使用所属职业前缀。
- 通用 `skillLogicId` / `talentLogicId` 处理器可以跨职业复用；独有机制使用明确的职业运行时处理器。

### 挑战先行与主线解锁

- 第一章喧鸣深窟永久只允许战士T，用作教程章节。
- 完成第一章后开放挑战1~3，并允许在这三关试用熊T。
- 分别使用熊T通过挑战1~3后，永久解锁熊T在第一章以外主线关卡中的使用权。
- 完成第二章后开放挑战4~6，并允许在这三关试用死亡骑士T。
- 分别使用死亡骑士T通过挑战4~6后，永久解锁死亡骑士T在第一章以外主线关卡中的使用权。
- 完成第三章后开放挑战7~9，该组三关预留给第四职业。
- 后续每个新增职业继续绑定独立的三关试炼组，例如挑战10~12、13~15；新增职业优先通过追加挑战组提前开放，不改写既有职业的解锁条件。
- 挑战中的职业资格累计开放：战士T永久可用，当前试炼职业临时可用，已永久解锁的旧职业继续可用。
- 胜利按 `(challengeId, classId)` 独立记录。其他职业的胜利不能替代目标职业的试炼进度，三次胜利不要求连续，职业永久解锁后不回退。

### 已选架构：职业注册表与进度策略服务

`player_build.xlsx` 继续作为职业战斗内容的策划来源，负责职业定义、技能、天赋、状态、默认构筑和图标映射。代码侧新增三个边界：

1. `PlayerClassRuntimeRegistry`
   - 保存资源模型、职业专属运行时处理器、AI 策略类型和界面排序等技术元数据。
   - 不重复保存策划表已提供的职业名称、技能描述和战斗数值。
2. `ClassTrialPolicy`
   - 每个非初始职业对应一个恰好包含三个不同挑战关卡的试炼组。
   - 记录 `classId`、三个 `challengeIds`、挑战组开放所需主线里程碑和主线解锁范围。
   - 新增职业通过追加策略项完成，不增加职业专用条件分支。
3. `StageClassAvailabilityService`
   - 根据模式、关卡、主线进度、试炼策略和永久解锁记录，统一计算当前关卡可选职业。
   - 关卡选择 UI、构筑入口、战斗入口和自动评测只消费该服务的结果，不各自实现职业资格判断。

建议存档边界：

- `challengeVictoriesByClass`：按挑战与职业保存胜利记录。
- `campaignUnlockedClassIds`：保存永久主线职业资格。
- `selectedClassId`：保存当前选择。
- `buildsByClassId`：按职业隔离玩家构筑。

### 已确认的关卡入口交互

- “进入这一关”按钮左置并使用固定宽高，文本为“职业名 进入这一关”。
- 右侧职业按钮使用固定 4 列 × 3 行网格，共预留 12 个位置；未显示职业仍保留布局空间。
- 职业按钮按注册顺序压紧，并按上、中、下、下一列顶部的纵向优先顺序排列。
- 可用职业按钮保持完整职业色；金色描边表示当前选择，绿色勾表示该职业已经通过本关。
- 每个职业图标都是独立按钮，点击后切换该关使用的职业。
- 不可用职业不显示；试炼组中的当前试炼职业视为临时可用。

## 已定稿基线：存档、通关记录与数据流

### 存档结构

职业扩展后的存档应至少包含：

```ts
{
  selectedClassId: 'warrior_t',
  buildsByClassId: {
    warrior_t: PersistedBuildState,
  },
  challengeVictoriesByClass: {
    druid_bear_t: ['Challenge-1', 'Challenge-2'],
  },
  campaignVictoriesByClass: {
    warrior_t: ['RingingDeeps-1'],
  },
  campaignUnlockedClassIds: ['warrior_t'],
}
```

- `selectedClassId` 保存玩家最近主动选择的职业偏好。
- `buildsByClassId` 按职业隔离主动技能与被动天赋构筑。
- `challengeVictoriesByClass` 按职业保存挑战通关记录，用于试炼解锁和挑战职业按钮的绿色勾。
- `campaignVictoriesByClass` 按职业保存主线通关记录，用于主线职业按钮的绿色勾。
- `campaignUnlockedClassIds` 保存永久获得的主线职业资格。
- 挑战与主线记录分开保存，避免挑战复用关卡模板时发生 ID 或语义混淆。

### 胜利结算

1. 战斗胜利提交 `mode + stageId/challengeId + classId`。
2. 进度服务幂等写入对应职业的通关记录。
3. 挑战胜利后，服务检查该职业自己的三关试炼是否全部完成。
4. 满足条件时，将该职业加入 `campaignUnlockedClassIds`；重复胜利和重复解锁不产生额外状态。
5. 关卡界面重新通过 `StageClassAvailabilityService` 计算可用职业和每个职业的通关勾。

### 旧存档迁移与容错

- 当前旧存档中的单一 `build` 迁移到 `buildsByClassId.warrior_t`。
- 旧玩家默认 `selectedClassId = warrior_t`，且 `campaignUnlockedClassIds` 至少包含 `warrior_t`。
- 已保存的永久职业资格不因后续试炼策略调整而删除。
- 加载时根据已有三关胜利补算遗漏的永久资格，并与已保存资格取并集。
- 最近选择的职业在当前关卡不可用时，只对本关临时回退为战士T，不覆盖原有偏好。
- 新职业没有保存构筑时，从该职业自己的默认构筑初始化，禁止继承或混入战士T技能与天赋。
- 存档加载、胜利写入、迁移和解锁必须保持幂等。

### 最低测试范围

- 旧存档迁移后保留战士T构筑和原有主线进度。
- 目标职业完成试炼2/3时不解锁，完成3/3时只解锁一次。
- 使用错误职业通过试炼关卡不计入目标职业解锁。
- 第一章始终只返回战士T，后续挑战正确累计已解锁职业和当前试炼职业。
- 当前职业不可用时正确临时回退，返回可用关卡后恢复原偏好。
- 各职业构筑、挑战胜利、主线胜利和绿色勾互不串用。

## 已定稿基线：主线累计解锁与挑战快照

新职业不在自己的三关试炼中单独渐进解锁技能和天赋。所有职业统一跟随主线进度累计解锁；挑战关卡保存其开放时对应章末关卡的完整构筑快照。

当前映射如下：

| 挑战组 | 开放里程碑 | 主动技能快照 | 被动天赋上限 | build规则 |
| --- | --- | --- | --- | --- |
| 挑战1~3 | `RingingDeeps-6` | 累计到第一章第6关，共5个战士T主动技能 | tier 1 | `standard_5slot` |
| 挑战4~6 | `WestFall-6` | 累计到第二章第6关，共10个战士T主动技能 | tier 2 | `8slot_0` |
| 挑战7~9 | `Zul'Aman-6` | 累计到第三章第6关，共16个战士T主动技能 | tier 3 | `8slot_2` |

规则：

- 同一挑战组三关使用完全相同的技能、天赋和 build 规则快照。
- 挑战表的 `unlockedActiveSkillIdsCsv` 保存截至对应章末的完整主动技能集合，不保存该关新增量。
- `passiveTalentUnlockTier` 显式保存对应章末被动天赋 tier。
- `challenge_stage_content.xlsx -> 关卡.buildRuleId` 与 `challenge_encounter_balance.xlsx -> 关卡开场.buildRuleId` 必须一致。
- `sourceStageIdsCsv` 仍只用于策划参考，运行时不得据此反推技能、天赋或 build 规则。
- 后续挑战10~12、13~15等新组，继续复制其开放章节第6关的完整快照。
- 所有可进入该挑战的职业共享同一进度档位；新增职业应提供与该档位槽位数、按键、点数预算等价的职业构筑规则。

### 职业无关构筑模板

`standard_5slot`、`8slot_0`、`8slot_1`、`8slot_2` 等 `buildRuleId` 定义进度档位的通用构筑模板，不再绑定某个职业：

- 构筑模板只负责总点数、主动槽位数、可用按键和继承策略。
- 当前 `selectedClassId` 单独限定可以进入构筑的主动技能与被动天赋；技能和天赋的 `classId` 必须与当前职业一致。
- 默认主动和被动构筑按 `(buildRuleId, classId)` 查找，禁止从其他职业回退。
- 构筑规范化接口必须显式接收 `classId`，并同时校验模板限制、职业归属和当前关卡解锁快照。
- `构筑规则定义.classId` 在新模型中不再作为必填职业绑定；实施时应改为可空的兼容字段，现有共享模板行应清空该值。
- `职业定义.recommendedBuildRuleIdsCsv` 可以让多个职业引用同一组通用模板。
- 如果某个职业未来确实需要特殊构筑预算，应新增显式的专用模板，而不是让通用模板根据职业偷偷变值。

2026-07-22 已定向同步：

- `public/designer-data/challenge_stage_content.xlsx`
  - 挑战1~3：5个主动技能、tier 1、`standard_5slot`。
  - 挑战4~6：10个主动技能、tier 2、`8slot_0`。
  - 挑战7~9：16个主动技能、tier 3、`8slot_2`。
- `public/designer-data/challenge_encounter_balance.xlsx`
  - 九关开场 `buildRuleId` 与上述三组快照一致。
- 本次没有修改 `allowedClassIdsCsv`；熊T和死亡骑士T尚未写入 `player_build.xlsx`，职业资格将在对应职业数据存在并进入实现阶段后同步。

## 已定稿基线：职业战斗内容与运行时边界

### 运行时职责

- `PlayerClassRuntimeRegistry` 只登记主资源、界面顺序、AI策略和职业运行时初始化器等技术能力，不重复保存策划表中的名称、技能和数值。
- 战斗入口显式接收 `classId + buildRuleId + classBuild`，并再次拒绝跨职业技能或天赋。
- 资源定义必须按 `classId` 精确命中；缺少职业注册时阻止进入战斗并报告错误，禁止静默回退战士T。
- 嘲讽、打断、控制、普通伤害、自疗、吸收和减伤继续复用通用 `skillLogicId`、`talentLogicId` 和状态处理器。
- 铁鬃、骨盾等职业状态使用职业前缀，但仍通过通用状态系统运行。
- 熊T一期通过怒气定义、技能处理器和状态组合闭合，不增加独立战斗循环。
- DKT符文槽等副资源进入带 `classId` 判别的 `classRuntime`，禁止塞入现有单一 `player.resource` 数值。
- 近期承伤窗口作为可复用的有效生命伤害记录，只统计减伤、吸收和免疫后真正扣除的生命；反射或被吸收伤害不计入死亡打击治疗基数。
- 职业独有逻辑通过注册表响应战斗事件，不在 `encounterFactory` 中累积职业条件分支。
- 伤害、治疗、吸收和资源变化继续进入统一战斗日志，并记录职业及技能或天赋来源。

### 内容规模

- 每个新职业主动技能目标约15~17个，章末累计开放节奏接近5个、10个、完整技能池。
- 每个新职业被动天赋不得少于20个。
- 四个被动 tier 优先采用接近平均的 `5 / 5 / 5 / 5` 分布；超过20个时也应尽量均匀分配。
- 数量允许因职业机制小幅调整，但同一章末必须形成完整坦克职责组合，不能依靠少量超强技能补足内容缺口。

### 失败与校验

- 缺少职业注册、默认构筑、技能或天赋处理器时，策划表校验或战斗入口必须明确失败。
- 技能、天赋、默认构筑、状态和运行时处理器中的跨职业引用必须被校验器拦截。
- 运行时不得以战士T定义作为新职业的隐式兜底。

## 已定稿基线：挑战先行平衡与自动评测

### 报告维度

- 自动报告以 `(stageId, classId, buildRuleId)` 为基本维度。
- 每个职业独立生成合法构筑、运行模拟和输出结果，禁止用跨职业最高通过率替代单职业结论。
- 报告同时展示有效承伤、治疗、吸收、仇恨稳定性、危险读条处理、队伍压力、资源断档、操作负载和学习路径。

### 稳定标准

- 新职业在自己的三关试炼中，学习型AI难度与战士T同关结果原则上不超过一个难度档位差异。
- 三关平均最佳通过率与战士T的绝对差异目标不超过15个百分点。
- 新职业某关学习型AI通过率低于5%，而战士T达到15%以上时，判定为职业工具缺口。
- 新职业某关达到85%以上，而战士T不超过69%时，判定为明显强度溢出。
- 允许熊T在持续物理与多目标场景、DKT在魔法与残局场景形成局部优势，但新职业不能在三关中全面优于战士T。
- 每个技能和天赋需要通过 delta 分析检查贡献，避免单个必选项决定绝大多数通过率，也避免大量无可测影响的选项。

### 评测与开放流程

1. 运行策划表校验和静态数值估算。
2. 按职业和关卡快照生成合法构筑。
3. 运行固定策略AI，检查默认构筑与基础可玩性。
4. 运行学习型AI，检查合理构筑与策略上限。
5. 运行技能、天赋及组合 delta 分析。
6. 使用推荐构筑进行人工三关试玩。
7. 达到稳定标准后，才允许该职业进入第一章以外主线。

- 自动评测保持只读，不得自动回写策划表。
- 新职业未达标时优先调整职业数据和AI策略；不得先修改主线、挑战敌人或关卡机制来迁就职业。

## 已定稿基线：实施分期与交付边界

### 阶段0：多职业基础设施

- 将构筑规则改为职业无关模板。
- 建立职业注册表、三关试炼策略和关卡职业资格服务。
- 升级存档并迁移现有战士T构筑。
- 实现按职业隔离的构筑、挑战/主线胜利记录和永久解锁。
- 完成固定12槽职业选择UI。
- 保证战士T行为和现有存档无回归；此阶段不加入未完成的新职业。

### 阶段1：熊T

- 在 `player_build.xlsx` 建立完整 `druid_bear_t` 数据，而不是只发布技术样例。
- 补齐15~17个主动技能、至少20个被动天赋、怒气参数、处理器、状态、图标和默认构筑。
- 挑战1~3开放熊T试用；开发期通过固定AI、学习型AI、delta分析和人工三关验证后，才进入可发布版本。
- 发布版本中，玩家使用熊T完成三关后永久解锁第一章以外主线资格。

### 阶段2：死亡骑士T

- 熊T闭环稳定后再开发 `dk_blood_t`。
- 先实现近期有效承伤窗口和符文/副资源容器，再添加完整职业内容。
- 挑战4~6作为DKT试炼，并沿用相同的独立职业评测和主线解锁流程。

### 阶段3：第四职业

- 挑战7~9保留第三章末构筑快照。
- 第四职业名称、ID和机制有意留到后续独立设计，不在本轮预设。
- 之后每个职业继续追加新的三关组，不修改已有试炼组。

### 非目标

- 第一章不兼容战士T以外职业。
- 不同时并行制作熊T和DKT完整内容。
- DKT第一版不复刻完整WoW六符文系统。
- 不通过修改主线或挑战敌人数据迁就职业强度。
- 不自动生成、覆盖或回写策划工作簿。
- 不允许缺失数据时回退到战士T。
- 不把开发期未通过评测的职业加入可发布构建。

正式设计规格：`docs/superpowers/specs/2026-07-22-player-tank-class-expansion-design.md`。

## 调研摘要

参考方向来自魔兽世界坦克专精的常见职责划分：

- 防护战士：盾牌、重甲、怒气、主动预判减伤。核心体验是提前开盾牌格挡、无视苦痛、盾墙，并用打断/控制处理危险读条。
- 守护德鲁伊：熊形态、高生命、高护甲、怒气驱动。核心体验是稳定承伤、铁鬃覆盖、狂暴回复自疗，操作压力低于战士但功能性更少。
- 鲜血死亡骑士：符文/符文能量、骨盾、死亡打击。核心体验是受伤后用资源把血线拉回来，容错来自自救，难点来自资源断档和技能时机。

## 与现有战士T的差异目标

不要把新职业做成换皮战士T。三个职业建议形成以下三角：

| 职业 | 核心防御 | 资源节奏 | 强项 | 弱项 |
| --- | --- | --- | --- | --- |
| 战士T | 预判格挡和吸收 | 怒气自然回复、受击、攻击获取 | 物理尖刺、反射、控制、打断 | 魔法/不可格挡压力、窗口判断错误 |
| 熊T | 高血量、高护甲、持续覆盖和自疗 | 怒气，偏稳定获取 | 稳定承伤、多目标仇恨、低操作门槛 | 精确反射/强打断少，爆发处理弱 |
| 死亡骑士T | 受伤后反打治疗、骨盾层数 | 符文/符文能量，建议分阶段引入 | 魔法工具、自救、残局能力 | 资源断档时波动大，AI 和玩家学习成本高 |

## 推荐一期：熊T

职业 ID 已定为：`druid_bear_t`。

资源：

- 一期复用 `rage` 资源定义，不新增第二套资源系统。
- 在 `src/game/encounter/playerResourceSystem.ts` 中增加该 `classId` 的资源参数。
- 熊T怒气建议更稳定、受击收益略低于战士，避免它既厚又能频繁爆发自疗。

主动技能草案：

| 技能 | 定位 | 建议逻辑 |
| --- | --- | --- |
| 裂伤 | 单体伤害/产怒 | 当前目标伤害，产生仇恨和怒气 |
| 痛击 | 多目标仇恨 | 范围伤害或周期流血，适合作为熊T多目标基础 |
| 铁鬃 | 核心主动减伤 | 消耗怒气，短时间提高物理减伤/护甲，可叠层或刷新 |
| 狂暴回复 | 核心自疗 | 消耗怒气，按最大生命或近期承伤治疗自己 |
| 树皮术 | 短冷却通用减伤 | 全伤害减免，操作要求低 |
| 生存本能 | 长冷却大减伤 | 大额全伤害减免，用于高危期 |
| 低吼 | 单体嘲讽 | 复用 `taunt` 或 `taunt_single` 模式 |
| 迎头痛击 | 打断 | 复用 `interrupt` 类逻辑 |
| 群体缠绕/乌索尔旋风 | 功能控制 | 控制或短时间降低敌人目标切换能力 |
| 化身 | 爆发窗口 | 提高怒气获取、伤害或铁鬃/痛击效率 |

被动天赋方向：

- 厚皮：提高最大生命或基础减伤。
- 强化铁鬃：提高铁鬃持续时间、减伤或允许额外层数。
- 狂野恢复：狂暴回复额外治疗或降低资源消耗。
- 血性狂乱：痛击/流血命中多个目标时获得怒气。
- 适者生存：降低生存本能冷却或提供第二层小减伤。
- 乌索克守护：团队压力高时获得额外减伤或自动回复。

熊T一期验收重点：

- 在 RD/WF/ZA 已有关卡中，熊T不应该全面优于战士T。
- 多目标普通伤害场景应更舒服。
- 需要精确反射或连续打断的场景应弱于战士T。
- 自动评分需要能分别输出战士T和熊T的 build/通过率，而不是只取全职业最优掩盖职业差异。

## 推荐二期：死亡骑士T

职业 ID 已定为：`dk_blood_t`。

一期资源实现建议：

- 不要一开始完整复刻 WoW 的 6 符文系统。
- 先实现：
  - 主资源：符文能量，`runic_power`。
  - 副资源：符文层数或冷却槽，最多 3 枚，每若干秒恢复 1 枚。
  - 近期承伤池：记录过去 5 秒玩家实际受到的伤害。
- 如果副资源系统代价过高，第一版可以把符文技能做成普通冷却，把死亡打击消耗符文能量，并先实现近期承伤池。

主动技能草案：

| 技能 | 定位 | 建议逻辑 |
| --- | --- | --- |
| 心脏打击 | 基础输出/资源 | 消耗符文或冷却，造成伤害并生成符文能量 |
| 骨髓分裂 | 骨盾维护 | 获得骨盾层数，层数提供护甲/减伤 |
| 死亡打击 | 核心自救 | 消耗符文能量，按近期承伤治疗，并可产生少量吸收 |
| 吸血鬼之血 | 血量/治疗窗口 | 提高最大生命和受到治疗效果 |
| 反魔法护罩 | 魔法工具 | 吸收或免疫部分魔法伤害，吸收成功时生成符文能量 |
| 死亡之握 | 仇恨工具 | 强制目标转向玩家，或短期提高玩家仇恨 |
| 枯萎凋零 | 区域仇恨 | 区域持续伤害/仇恨 |
| 符文刃舞 | 爆发防御 | 短时间提高招架/减伤/资源效率 |

被动天赋方向：

- 白骨壁垒：骨盾层数提高减伤。
- 赤色饥渴：死亡打击有效治疗降低大技能冷却。
- 饮血者：周期伤害转化少量治疗。
- 符文分流：低血线自动触发小减伤或消耗资源减伤。
- 反魔法专精：反魔法护罩提高吸收或返还资源。
- 腐烂壁垒：枯萎凋零内敌人对玩家造成伤害降低。

死亡骑士T验收重点：

- 近期承伤池必须只统计玩家实际承受的生命伤害，不应把被吸收、免疫、反射掉的伤害也算进死亡打击治疗基数。
- 高通过率不等于低难度。自动报告应把 DK 的学习路径、技巧要求和操作负载作为独立维度展示。
- 需要新增 AI 职业策略：死亡打击应在受伤后、血线危险或近期承伤池较高时释放，而不是 CD 好了就用。

## 实现路线建议

### Step 1：职业选择和构筑边界

职业选择边界已经确认：挑战模式先行，完成职业对应的三关试炼后，永久解锁该职业在第一章以外主线中的使用权。

需要检查：

- `StageSelectScreen` 是否按关卡/模式列出职业。
- 保存数据是否按 `classId` 保存构筑。
- 默认构筑是否能按 `buildRuleId + classId` 正确回退。

### Step 2：资源系统扩展

熊T：

- 在 `PLAYER_RESOURCE_DEFINITIONS` 增加 `druid_bear_t`。
- 保持 `resourceCost` 字段复用。

死亡骑士T：

- 至少增加 `dk_blood_t` 的主资源定义。
- 如果要实现副资源，不要塞进单个 `player.resource` 数值里；应新增明确 runtime 字段，例如 `classRuntime`，并写迁移/初始化测试。

### Step 3：主动技能逻辑

优先复用现有处理器：

- 嘲讽：`taunt` / `taunt_single`
- 打断：现有 interrupt 处理器
- 控制：`stun` 类处理器
- 普通伤害：可参考 `revenge` / `shield_slam` / `thunderstruck`
- 自身减伤：可参考 `shield_wall` / `shield_block`
- 吸收：可参考 `ignore_pain`

新增处理器时要满足：

- 读取 `主动技能效果` 的 `valueA/valueB/durationMs/statusId/targetSelector/threatDelta/threatMultiplier`。
- 写入战斗统计：伤害、治疗、吸收、队伍承伤、坦克承伤不能丢。
- 和 AI 策略可读的标签对齐，例如 `survival`、`interrupt`、`threat`、`heal`、`aoe`。

### Step 4：被动天赋逻辑

新增天赋优先做成以下类型：

- 改数值型：最大生命、伤害减免、资源回复、技能冷却。
- 改技能型：增强某个主动技能。
- 触发型：低血、成功打断、受到伤害、造成治疗后触发。
- 队伍型：降低压力、提高队伍治疗、降低队伍承伤。

不要把职业专属被动硬塞进战士T现有逻辑分支。新 `talentLogicId` 应在注册表里明确命名。

### Step 5：自动评测和报告

新增职业后必须更新：

- build 生成器：确保 `classId` 不混用战士技能和熊T/DK技能。
- 学习型 AI：为熊T/DK加入对应技能使用偏好。
- 数值估算：按职业输出资源获取、承伤、治疗、吸收、伤害。
- 手动测试记录：`manual_playtest_builds.xlsx` 后续应能按职业记录推荐 build。

## AI 策略初稿

熊T策略：

- 保持铁鬃覆盖，尤其在物理高压窗口。
- 血量低或近期承伤高时使用狂暴回复。
- 多目标时优先痛击保持群体仇恨。
- 大压力窗口使用树皮术/生存本能。
- 打断优先级低于战士，但关键读条仍要处理。

死亡骑士T策略：

- 骨盾低层数时补骨髓分裂。
- 近期承伤池高或血线危险时使用死亡打击。
- 魔法高危技能前使用反魔法护罩。
- 资源不足时保留死亡打击资源，不要把符文能量全部打成输出。
- 低血线时优先自救，不优先追伤害。
