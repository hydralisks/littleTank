# 难度自动评分系统

本系统是只读分析工具：它评估关卡难度并输出报告，但不得修改 `public/designer-data/` 下由策划维护的工作簿。

## 评分输入范围

允许参与定级的输入：

- 不同操作 profile 下的通过率。
- 不同反应延迟和失误率下的通过率。
- 当前关卡合法 build 变体下的通过率，包含主动技能与被动天赋。
- 静态数值风险，仅作为早期提示层。
- Build 宽容度修正：只在“空余 4 个或更多技能点的 build 仍达到最优 build 通过率一半以上”时，下调一个难度档位。

明确不参与定级的输入：

- 玩家剩余血量比例。
- 队伍剩余血量比例。
- 队伍剩余压力比例。
- 任何其它战斗结束时的间接舒适度数值。

这些间接数值可以用于调试 trace，但不能决定关卡是 `balanced`、`hard`、`expert` 或其它标签。

## 三层评分

| 层级 | 实现 | 用途 |
| --- | --- | --- |
| 静态数值评分 V3 | `src/game/balance/staticStageScoring.ts` | 不跑战斗，拆分不可处理压力、可处理压力、敌方表面/有效支援、玩家工具抵消和执行复杂度。它用于发现表面结构风险并解释静态与模拟差异，不替代模拟和人工测试。 |
| 固定策略 AI | `src/game/balance/balanceSimulator.ts` | 两段式 build 搜索，然后用统一战术假设跑 average/skilled/expert profile。固定策略会按目标仇恨、敌方读条/技能循环风险、低血量斩杀价值、irregular 周期补仇恨和坦克承压分摊来选择目标。 |
| 学习型 AI | `src/game/balance/learningBalanceEvaluator.ts` | 先探索合法 build、读条处理策略与战术策略，再收敛到共享胜率较高的 build/策略集合进行正式样本。 |

`npm run analyze:balance` 会在内存中加载当前策划工作簿后运行三层评分。

2026-06-03 起，固定策略 AI 的最终样本和学习型 AI 的最终样本会额外采集内部战斗诊断摘要。诊断摘要来自 `buildEncounterStats` 与 `buildCombatDiagnostics`，再由 `src/game/balance/diagnosticScenarioSummary.ts` 聚合为 scenario 级信号出现率、失败样本均值、置信度和候选 action。策划版自动评分 Markdown 不展示这些失败原因明细，只保留“每个关卡难度是否符合预期”的结论，以及静态评分、固定策略 AI、学习型 AI 三类玩家梯度参考；结构化诊断数据保留在 `reports/balance/*.json`，供后续内部自动评测和学习型 AI 策略扩展使用。该层不进入玩家结算 UI，也暂时不改变 AI 决策。

## 静态评分 V3 口径

2026-05-27 起，静态评分不再把所有敌方技能状态简单线性加总，而是输出以下组件：

- `rawThreatScore`：敌人数量、总血量、伤害/压力读条、长线压力、目标复杂度、操作负载和机制风险组成的原始威胁。
- `enemySupportRisk`：敌方治疗、增益和自保类技能的支援风险。鱼人治疗这类 `mostInjured` 读条会按可打断、读条窗口、目标锁定后的过量治疗可能进行折扣，不再按每次满额治疗计入。
- `toolCoverageScore`：玩家当前主动技能、自动伤害、自动治疗、嘲讽、打断、控制、防御和范围能力对关卡需求的覆盖。
- `adjustedThreatScore`：最终静态分。低强度支援风险只展示为诊断信息；超过低强度阈值的部分才小比例进入最终分，避免第一章小额治疗/自保破坏人工基线，同时让第二章多鱼人支援结构被看见。

静态评分 V3 的重点是解释“表面风险中哪些是真的压力、哪些能被当前工具处理、哪些只是执行复杂度”，不是给出最终难度结论。第二章人工测试后，应优先对比 `effectiveSupportRisk`、优先打断建议和模拟通过率，而不是只看最终静态标签。

V3 新增组件：
- `unavoidablePressureScore`：环境压力、长线压力和不可打断高影响读条。
- `answerablePressureScore`：可被打断、控制、嘲讽、防御或目标优先级处理的压力。
- `toolMitigationScore`：当前关卡已解锁工具对可处理压力的抵消。
- `effectiveSupportRisk`：考虑打断窗口、长读条、`mostInjured` 过量治疗和目标锁定后的有效支援。
- `executionComplexityScore`：多目标、多读条、站位分散和特殊仇恨带来的执行成本。

## 战术策略覆盖

固定策略 AI 当前覆盖以下通用人类打法假设：

- 高威胁击杀：优先处理当前读条风险高、技能循环伤害/压力高、且血量较低的敌人。
- Irregular 维护：对 `irregular` 仇恨类型敌人周期性补目标或嘲讽，但在坦克承压很高时允许短暂漏给队伍分摊。
- 关键技能保留：强 profile 会优先把打断/控制留给高影响读条和引导阶段。
- 防御资源铺开：如果盾墙等减伤已经生效，AI 不会立刻重复消耗同类防御资源。

学习型 AI 在上述基础上额外搜索战术策略：

- `strict-threat`：严格抢回丢失仇恨。
- `kill-high-impact`：更重视优先击杀高威胁目标。
- `allow-irregular-leak`：坦克高压时允许 irregular 目标短暂攻击队伍。
- `mechanic-focus`：提高带机制特征目标的优先级。

这些仍是启发式策略，不等同于真正理解所有机制。像 RingingDeeps-6 中“蜡像”和“暗影之锄”这样的状态链，后续需要把“状态来源和后续技能目标之间的关联假设”纳入学习型 AI 搜索。

当前机制链处理已加入第一步：AI 会识别 `shadow_hoa` 的后续目标应尽量打在已有 `waxed` / `waxed_p` 的一侧，避免把它当成普通丢仇恨问题立刻抢回。但这还不是完整学习型机制推理；后续仍需要把“让前置技能故意打哪一侧以服务后续技能”作为可搜索策略维度。

## AI 是否考虑被动天赋

已确认：AI 的 build 搜索包含被动天赋。

- `generateStageBalanceBuilds` 会读取 `getPassiveTalentUnlockTierForStage(stage)`，只生成当前关卡已解锁 tier 内的合法被动天赋组合。
- 每个候选 build 都是完整 `PersistedBuildState`，包含 `loadout` 与 `passiveTalentIds`。
- `runBalanceScenario` 创建战斗时直接把完整 build 传给 `createInitialEncounterState`，因此被动天赋会进入实际战斗模拟。
- 最佳 build 报告会输出 `passiveTalentIds`，便于检查 AI 选中的被动天赋。

## 标签标准

| 条件 | 标签 |
| --- | --- |
| 所有测试 profile/build 都没有有效通关 | `impossible` |
| expert `<20%` 且 skilled `<10%` | `near_impossible` |
| average `>=95%` | `trivial` |
| average `>=80%` | `easy` |
| average `>=55%` | `balanced` |
| average `>=25%` 或 skilled `>=55%` | `hard` |
| expert `>=35%` 或 skilled `>=20%` | `expert` |
| 其它情况 | `near_impossible` |
| 空余 `>=4` 技能点的 build 达到最优 build 通过率一半以上 | 在基础标签上最多下调一档 |

## 学习型 AI 独立评分

学习型 AI 现在使用单一较强玩家 profile，不再按 `average / skilled / expert` 三档解释结果。它的通过率代表“玩家反复尝试后找到较强 build 和策略”的表现，因此阈值比固定策略 AI 更严格。

| 学习型 AI 最佳通过率 | 难度标签 |
| --- | --- |
| `>=85%` | `trivial` |
| `70%-84%` | `easy` |
| `55%-69%` | `balanced` |
| `35%-54%` | `hard` |
| `15%-34%` | `expert` |
| `5%-14%` | `near_impossible` |
| `<5%` | `impossible` |

学习型 AI 有单独的 build 宽容度修正：如果至少 3 个 build 达到 `max(65%, 最优通过率 * 80%)`，说明玩家不需要找到唯一精确解法，学习型难度下调一档。这个规则不使用固定策略 AI 的“空余 4 技能点”条件。

当前诊断层对学习型 AI 的作用是“候选 action 输出”而不是“即时策略控制”。例如 `high_danger_casts_completed` 会映射到高危读条处理优先级候选，`pressure_concentrated` 会映射到压力来源处理候选。后续如果要扩展学习型策略，应先让这些候选进入搜索空间，再继续用通过率筛选，避免诊断层直接覆盖模拟结果。

## 学习技巧需求

学习技巧需求不是难度标签，而是回答“玩家需要学会多少本关专属技巧”。它使用学习型 AI 的探索样本和最终样本计算：

- 默认 build 与最佳非默认 build 的差距。
- 是否依赖特殊读条策略，例如末端打断或处理中低危读条。
- 是否依赖特殊战术策略，例如允许 irregular 分摊、机制目标优先、蜡像与暗影之锄目标链。
- 最终可行 build 数量是否狭窄。

| 技巧需求分 | 标签 | 含义 |
| ---: | --- | --- |
| `0-19` | `none` | 基本沿用常规打法 |
| `20-39` | `basic` | 需要轻微调整目标或打断时机 |
| `40-59` | `moderate` | 需要明确学习本关节奏或 build 调整 |
| `60-79` | `high` | 需要掌握关卡机制级技巧 |
| `80+` | `specialist` | 接近特解，需要专门策略 |

## 学习型 AI 操作负载修正

2026-05-25 后续校准中发现：学习型 AI 的“单次操作固定失误率”不能很好表达真实玩家学习后的表现。玩家面对 RD3/RD4 这类“少数明确窗口”的关卡时，只要学会优先目标和关键读条，执行成本会下降；而 RD5/RD6 以及未来 8 键关卡中，技能数量、目标数量、GCD/资源冲突、机制链和需要保留的关键技能会让复合操作成本上升。

因此学习型 AI 需要额外输出并使用“操作负载”信息：

- **清晰窗口收益**：敌人少、主动键位少、最终策略不依赖特殊机制链时，学习后的玩家更容易稳定执行，学习型难度可以下调一档。
- **复合操作惩罚**：敌人多、主动键位多、非默认 build 明显领先、特殊读条/irregular/机制链收益明显时，学习型难度可以上调一档。
- 该修正最多上下调整一档，避免覆盖通过率本身。
- 技巧需求仍然独立展示；操作负载用于解释“学会之后是否容易执行”，技巧需求用于解释“需要学会什么”。

这个设计的目标是让 RD3/RD4 这类清晰教学窗口更接近人工测试，同时为后续 8 键开放关卡预留更高的注意力成本。

## 今日规则变更

2026-05-24 起，移除旧的“多种 build 都能通关则下调难度”规则。新的 build 宽容度修正只保留一个条件：

> 如果存在空余 4 个或更多技能点的合法 build，并且该 build 的通过率不低于最优 build 通过率的一半，则最终难度下调一档。

同时，评分总表需要单独展示“达到最优 build 一半以上通过率的 build 数量”，并额外标出其中“空余 4 点以上”的数量。

## 2026-05-26 静态评分拓维与策划建议输出

本轮继续保持只读原则：评分系统只读取 `public/designer-data/*.xlsx`，不自动修改策划表。任何数值调整仍应由策划在表中执行后重新导入。

静态评分从旧的“敌人数量 + 总血量 + 平均技能风险”扩展为以下维度：

- 敌人规模：敌人数量与总生命值。
- 读条风险：平均技能风险与危险读条密度，避免把低危高频读条直接等同于高危窗口。
- 目标结构：敌人类型数量、技能多样性、站位分散程度、`irregular` / `bloodlust` 仇恨类型。
- 操作负载：高危技能数量、当前可用打断/控制/防御工具覆盖、四目标以上多线监控成本。
- 机制风险：治疗/队伍目标/自身增益/引导/不可打断技能、开场状态、关卡 affix 和特殊规则。
- 工具覆盖：主动技能、自动攻击、自动治疗、群体工具、嘲讽、打断、控制和防御技能是否覆盖本关需求。
- 长线压力：基于总血量、自动伤害和环境压力估算战斗拖长后的压力风险。

这套静态评分已用当前 RingingDeeps 人工基线重新校准：RD1/RD2 为 `easy`，RD3/RD4 为 `hard`，RD5 为 `balanced`，RD6 为 `expert`。它仍然不是最终判定，只用于更早发现“表面数值或结构可能有风险”的关卡。

学习型 AI 做了一个小范围校准：当关卡属于低技巧需求、窗口清晰、玩家学会后执行成本较低的情况时，如果原始通过率落在 `near_impossible` 低段，可以在最终学习型难度中额外下调到更接近人工体验的区间。这个修正主要用于解释 RD3/RD4 这类“少数明确操作窗口”关卡，不用于掩盖 RD5/RD6 或未来 8 键关卡的复合操作成本。

新增 `src/game/balance/balanceRecommendation.ts`，用于把三层评测合成为给策划看的建议。建议只输出方向，不改表，类型包括：

- 原始数值偏高/偏低：三层评测同时偏离人工目标时，建议调整总生命值、关键敌人血量、读条窗口或伤害。
- 机制/执行负载偏高：学习型 AI 显示技巧需求或操作负载过高时，建议增加机制容错或补教学，而不是只改伤害。
- AI 策略覆盖差异：固定策略 AI 和学习型 AI 差距过大时，优先检查 AI 是否缺少策略，或本关是否需要教学提示。
- 构筑依赖较窄：只有极少 build 达标时，建议检查默认 build、技能解锁提示或替代技能容错。
- 静态与模拟分歧：静态评分和 AI 通过率差距较大时，作为公式或 AI 行为覆盖的诊断信号。

`npm run analyze:balance` 的章节报告会新增“数值与敌人设计建议”表，显示每关的自动信号、严重度、置信度和中文建议。该表面向策划参考，不能直接替代人工测试结论。

## 2026-05-26 静态战术建议与策略提示

为了让静态评分、固定策略 AI 和学习型 AI 共用一套更稳定的战术依据，本轮又补了两类只读输出：

- **优先击杀目标**：按敌人平均技能威胁、当前剩余血量、击杀成本和仇恨类型综合排序。目标不是“血少就一定先打”，而是优先选出“清掉后能最快降低敌方输出、并且击杀成本合理”的敌人。
- **优先打断目标**：按敌方读条/引导的伤害、压力、状态收益和是否影响输出节奏来排序。像 RD4 这类关卡，学徒的高压飞弹和地卜师的减速都应该被看见，但排序应由收益驱动，而不是写死关卡名。

这两类结果会进入自动评分输出，也会作为固定策略 AI 与学习型 AI 的参考输入。后者不再直接硬编码某个关卡专属战术，而是先消费静态建议，再在此基础上做搜索和筛选。

同时，`stage_content.xlsx` 的关卡表新增了可选中文字段 `strategyTips`：

- 这是策划写给 AI 的本关特殊技巧提示。
- 它不改数值，只用于提示学习型 AI 哪些机制链或目标联动值得尝试。
- 例如蜡像与暗影之锄这类机制链，只会在关卡机制或 `strategyTips` 明示相关时进入学习型战术搜索；不会因为全局策略列表里有这类打法，就默认带到所有关卡。

这套设计的目的，是把“静态数值判断”“固定策略模拟”“学习型玩家模拟”三层结果尽量统一到同一套战术语言上，同时避免把特殊连招写成全局硬编码。

## 2026-05-30 Delta Analysis 与采样可信度

`npm run analyze:delta` 是 `npm run analyze:balance` 的只读配套工具：

- `analyze:balance` 回答“关卡整体难度是什么”。
- `analyze:delta` 回答“指定 build、被动天赋或组合相对基线贡献了多少通过率”。
- quick delta 样本只用于筛选候选。报告中标记为 `low` confidence 的结论，需要用 `--sample=standard` 或 `--sample=full` 复跑后，才适合作为调参建议。

示例：

```bash
npm run analyze:delta -- --stages=WestFall-2,WestFall-3 --type=passive --sample=quick
```

输出：

- `reports/balance/delta-analysis.md`
- `reports/balance/delta-analysis.json`

每条对照会输出 baseline 通过率、variant 通过率、绝对 delta、相对 delta、confidence、verdict 和复核建议。该工具仍然只读取 `public/designer-data/*.xlsx`，不会生成或覆盖策划表。
