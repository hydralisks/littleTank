# 玩家坦克职业扩展设计规格

日期：2026-07-22

状态：已批准，等待书面规格复核后进入实施计划

## 1. 目标

在现有战士T框架上建立可持续扩展的多坦克职业系统，并依次开发熊T与死亡骑士T。新职业先在独立挑战中完成教学、平衡和解锁，再进入第一章以外主线。

本设计同时解决：

- 职业身份、前缀和公共状态边界。
- 三关试炼、永久主线资格和逐职业通关记录。
- 多职业构筑、存档迁移和关卡入口交互。
- 主线累计技能/天赋解锁及挑战章末快照。
- 职业运行时、AI和自动评测扩展边界。
- 熊T、DKT和后续职业的交付顺序。

## 2. 职业身份与内容所有权

规范职业 ID：

- 战士T：`warrior_t`。
- 熊T：`druid_bear_t`。
- 死亡骑士T：`dk_blood_t`。
- 第四职业有意不在本设计中命名，后续单独设计。

职业专属技能、天赋和状态使用所属职业前缀，例如 `druid_bear_t_*`、`dk_blood_t_*`。语义完全一致的公共状态只保留一个规范定义，例如 `taunted`、`countered`、`stunned`。

`public/designer-data/player_build.xlsx` 是职业名称、技能、天赋、状态、默认构筑和图标映射的策划来源。代码不得重复保存这些策划数值。

## 3. 核心架构

### 3.1 PlayerClassRuntimeRegistry

职业运行时注册表只保存技术元数据：

- `classId`。
- 主资源定义。
- UI显示顺序与职业按钮图标键。
- AI策略 ID。
- 可选的职业运行时初始化器。

资源和职业查询必须精确命中。缺少注册项时阻止进入战斗并报告错误，禁止静默回退到战士T。

### 3.2 ClassTrialPolicy

每个非初始职业对应一个三关试炼组。策略项至少包含：

- `classId`。
- 三个不同的 `challengeIds`。
- 挑战组开放所需的主线里程碑。
- 永久主线资格范围。

当前策略：

- `druid_bear_t`：挑战1~3，完成第一章后开放。
- `dk_blood_t`：挑战4~6，完成第二章后开放。
- 第四职业：挑战7~9，完成第三章后开放。

未来职业使用挑战10~12、13~15等新组，不改写已有策略。

### 3.3 StageClassAvailabilityService

一个服务统一计算指定关卡的可用职业，UI、构筑入口、战斗入口和评测不得各自复制资格逻辑。

规则：

- 战士T永久可用。
- 第一章喧鸣深窟永久只允许战士T。
- 当前三关试炼允许临时使用对应试炼职业。
- 已永久解锁的职业在后续挑战中累计可用。
- 第一章以外主线允许战士T和已永久解锁职业。
- 其他职业的挑战胜利不能替代目标职业的试炼进度。
- `allowedClassIdsCsv` 表示关卡内容已经支持的职业上限；非空时，有效职业集合为该上限与玩家进度资格的交集。
- 新职业数据完整落地时，必须先把该职业加入对应试炼及后续挑战的 `allowedClassIdsCsv`，资格服务不能绕过内容支持上限。

## 4. 职业解锁与胜利数据流

胜利按模式、关卡和职业独立记录。挑战胜利流程：

1. 战斗胜利提交 `mode + challengeId + classId`。
2. 进度服务幂等写入该职业的挑战记录。
3. 服务检查该职业对应三关是否全部完成。
4. 完成后将职业加入永久主线资格，重复结算不产生额外变化。
5. 关卡界面重新计算可用职业和通关标记。

三关胜利不要求连续，职业永久解锁后不回退。

## 5. 存档与迁移

目标存档概念：

```ts
{
  selectedClassId: 'warrior_t',
  buildsByClassId: {
    warrior_t: PersistedBuildState,
  },
  challengeVictoriesByClass: {
    druid_bear_t: ['Challenge-1', 'Challenge-2'],
  },
  campaignVictoriesByClass: {
    warrior_t: ['RingingDeeps-1'],
  },
  campaignUnlockedClassIds: ['warrior_t'],
}
```

迁移规则：

- 旧存档的单一 `build` 迁移到 `buildsByClassId.warrior_t`。
- 旧玩家默认选择并永久拥有战士T。
- 已保存的永久资格不因后续策略修改而删除。
- 加载时将已保存资格与根据现有三关记录补算的资格取并集。
- 当前偏好职业在本关不可用时只临时回退战士T，不覆盖原偏好。
- 新职业缺少存档构筑时从自己的默认构筑初始化，禁止继承其他职业内容。
- 加载、迁移、胜利记录和资格补算均需幂等。

## 6. 职业无关构筑模板

`standard_5slot`、`8slot_0`、`8slot_1`、`8slot_2` 等规则是进度档位模板，只定义：

- 总构筑点数。
- 主动技能槽位数。
- 可用按键。
- 构筑继承策略。

职业合法性由 `selectedClassId` 单独决定：

- 主动技能和被动天赋的 `classId` 必须匹配当前职业。
- 默认构筑按 `(buildRuleId, classId)` 查找，不跨职业回退。
- 构筑规范化显式接收 `classId`，同时校验模板、职业和关卡解锁快照。
- `构筑规则定义.classId` 改为可空兼容字段；共享模板行清空职业绑定。
- 多个职业可在 `recommendedBuildRuleIdsCsv` 中引用同一组模板。

职业确需不同预算时新增显式专用模板，不在共享模板中按职业暗改值。

## 7. 主线累计解锁与挑战快照

新职业技能和天赋完全跟随主线累计解锁，不建立试炼专用解锁路线。

| 挑战组 | 开放里程碑 | 主动技能快照 | 被动tier | build规则 |
| --- | --- | --- | --- | --- |
| 挑战1~3 | `RingingDeeps-6` | 对每个可用职业累计到第一章末 | 1 | `standard_5slot` |
| 挑战4~6 | `WestFall-6` | 对每个可用职业累计到第二章末 | 2 | `8slot_0` |
| 挑战7~9 | `Zul'Aman-6` | 对每个可用职业累计到第三章末 | 3 | `8slot_2` |

同组三关使用完全相同的快照。`unlockedActiveSkillIdsCsv` 保存所有可用职业在该里程碑下的完整合法技能集合，不保存新增量。`passiveTalentUnlockTier` 显式保存章末 tier。

`challenge_stage_content.xlsx -> 关卡.buildRuleId` 与 `challenge_encounter_balance.xlsx -> 关卡开场.buildRuleId` 必须一致。`sourceStageIdsCsv` 只作策划参考，运行时不得据此推导解锁或规则。

当前挑战表只包含战士T技能和职业资格，因为熊T与DKT尚未写入 `player_build.xlsx`。新增职业数据时，同步扩充对应挑战组及之后挑战组的完整技能集合和 `allowedClassIdsCsv`。

## 8. 关卡入口UI

挑战和第一章以外主线使用相同职业选择交互：

- 左侧“进入这一关”按钮固定为约 `240 x 82px`。
- 右侧为4列3行、共12槽的固定职业网格。
- 单个职业按钮约 `50 x 50px`，间距约 `7px`；网格约 `221 x 164px`，三行总高约为主按钮两倍。
- 未显示职业使用不可见占位保留空间。
- 可用职业按注册顺序压紧，以上、中、下、下一列顶部的纵向优先顺序排列。
- 点击职业按钮切换本关职业，主按钮文本变为“职业名 进入这一关”。
- 职业按钮始终保持完整职业色。
- 金色描边表示当前选择。
- 绿色勾覆盖在按钮上，表示该职业已通过当前关卡。
- 不可用职业不显示；当前试炼职业视为临时可用。
- 第一章只显示战士T。

## 9. 战斗运行时边界

战斗入口显式接收 `classId + buildRuleId + classBuild`，并拒绝跨职业内容。

通用能力继续复用现有注册体系：

- 嘲讽、打断和控制。
- 普通伤害与仇恨。
- 自疗、吸收和减伤。
- 通用主动/被动状态。

熊T一期组合怒气定义、通用状态和必要技能/天赋处理器，不建立独立战斗循环。

DKT的符文槽等副资源进入带 `classId` 判别的 `classRuntime`，禁止塞入现有单一 `player.resource`。近期承伤窗口作为可复用能力，只统计减伤、吸收和免疫后真正扣除的生命；反射和被吸收伤害不进入死亡打击治疗基数。

职业独有机制通过注册表和事件处理器接入，不在 `encounterFactory` 中累积职业判断。所有伤害、治疗、吸收和资源变化继续进入统一战斗日志，并保留职业、技能或天赋来源。

## 10. 职业内容规模

- 每个新职业主动技能目标约15~17个。
- 章末累计主动技能节奏接近5个、10个、完整技能池。
- 每个新职业被动天赋不得少于20个。
- 四个被动 tier 优先采用 `5 / 5 / 5 / 5`；超过20个时仍近似均匀分布。
- 每个章末快照必须具备完整坦克职责组合，不能依赖少量超强技能补足内容缺口。

## 11. 平衡与自动评测

报告以 `(stageId, classId, buildRuleId)` 为基本维度。每个职业独立生成构筑、运行模拟和输出结果，禁止用跨职业最高结果替代单职业结论。

稳定标准：

- 新职业三关学习型AI难度与战士T同关原则上不超过一个档位。
- 三关平均最佳通过率绝对差目标不超过15个百分点。
- 新职业某关低于5%、战士T达到15%以上时，判定职业工具缺口。
- 新职业某关达到85%以上、战士T不超过69%时，判定明显强度溢出。
- 允许符合职业定位的局部优势，但新职业不能三关全面优于战士T。
- 技能和天赋需进行 delta 分析，拦截绝对必选项和大量无影响选项。

报告至少展示：有效承伤、治疗、吸收、仇恨稳定性、危险读条处理、队伍压力、资源断档、操作负载和学习路径。

评测顺序：策划表校验与静态估算、合法构筑生成、固定AI、学习型AI、delta分析、人工三关试玩。自动评测只读，不回写策划表。职业未达标时优先调整职业数据和AI策略，不修改关卡来迁就职业。

## 12. 错误处理与校验

以下情况必须在策划表校验或进入战斗前明确失败：

- 职业缺少运行时注册。
- 职业缺少自己的默认构筑。
- 技能或天赋缺少逻辑处理器。
- 构筑包含其他职业的技能或天赋。
- 职业专属状态使用错误前缀。
- 试炼组不是恰好三个不同挑战。
- 挑战关卡与开场的 build 规则不一致。

任何情况都不得静默回退到战士T。

## 13. 测试范围

基础设施测试：

- 旧存档迁移和重复迁移。
- 按职业隔离构筑与胜利记录。
- 2/3不解锁、3/3解锁、错误职业不计入、重复胜利幂等。
- 第一章强制战士T、后续挑战累计资格、主线资格过滤。
- 当前偏好临时回退但不被覆盖。
- 职业无关模板与按职业技能/天赋过滤。
- UI 12槽、纵向优先、选择描边和通关勾。

职业运行时测试：

- 精确资源注册且无战士T回退。
- 通用技能和状态跨职业复用。
- 职业专属机制和状态前缀。
- DKT有效承伤窗口排除吸收、免疫与反射。
- 战斗日志和评测按职业归因。

每个职业发布前还需通过完整数据校验、构筑生成测试、固定/学习AI、delta分析和人工三关试玩。

## 14. 实施分期

### 阶段0：多职业基础设施

完成职业无关构筑模板、职业注册表、试炼策略、资格服务、存档迁移、多职业构筑、胜利记录和12槽UI。战士T行为与旧存档必须保持兼容，不加入未完成职业。

### 阶段1：熊T

一次性建立完整 `druid_bear_t` 职业数据、怒气参数、技能/天赋处理器、状态、图标、默认构筑、AI策略和挑战1~3评测闭环。开发期达标后才进入可发布构建；发布后由玩家三关胜利永久解锁主线资格。

### 阶段2：死亡骑士T

熊T稳定后实现 `dk_blood_t`，先补近期承伤和副资源基础，再完成职业内容、AI和挑战4~6闭环。

### 阶段3：第四职业

挑战7~9保留第三章末快照。职业名称、ID和机制留到后续独立设计。之后继续追加新三关组。

## 15. 非目标

- 让第一章兼容新职业。
- 同时并行制作熊T和DKT完整内容。
- DKT第一版复刻完整WoW六符文系统。
- 修改主线或挑战敌人数据来迁就职业。
- 由运行时或自动评测自动生成、覆盖或回写策划工作簿。
- 运行时使用战士T作为新职业兜底。
- 将未通过挑战评测的职业加入可发布构建。

## 16. 已完成的策划表同步

2026-07-22 已按用户授权定向更新：

- `challenge_stage_content.xlsx`：挑战1~3、4~6、7~9分别采用第一、第二、第三章末主动技能、被动tier和 build 规则快照。
- `challenge_encounter_balance.xlsx`：九关开场 `buildRuleId` 与挑战关卡表一致。

没有运行任何 `public/` 生成器。熊T和DKT职业资格及技能集合将在对应职业数据实际加入 `player_build.xlsx` 时同步。
