# 关卡选择可见性与桌面入口布局设计

日期：2026-07-22

## 目标

在不修改既有职业试炼和主线解锁规则的前提下，收紧挑战模式的可见性，并调整桌面/平板端主线职业入口，使完整三行职业图标位于地图框内。

## 挑战模式可见性

- 挑战关卡仍由 `isChallengeStageOpen(stageId, highestClearedStageIndex)` 判断是否开放。
- 挑战列表只渲染当前已开放的挑战，不显示禁用卡片或未解锁占位。
- 第一组挑战尚未开放时，不渲染“挑战模式”按钮。
- 第一章完成、挑战1~3开放后显示“挑战模式”按钮；后续完成第二章、第三章时，挑战4~6、7~9继续累计显示。
- 如果外部状态要求直接打开挑战模式，但当前没有开放挑战，则界面回退到主线模式，禁止展示使用主线 fallback 拼出的伪挑战内容。
- 已进入挑战模式后，玩家选择的挑战必须来自过滤后的开放列表。

## 桌面与平板布局

- 本轮只验收宽度大于 `680px` 的桌面/平板布局。
- 保持 CTA `240×82`、职业网格4列×3行、槽位 `50×50`、图标约 `34×34` 和纵向优先排列不变。
- 主线 CTA 与职业网格作为同一个入口控件整体上移。
- 职业网格底部与地图框下边界保留约12px间距，确保第三行完整显示在地图框内。
- 不改变挑战面板已有的入口留白方式。
- 手机端本轮不作为验收目标，保留现有 CSS 映射；后续在不复制职业选择状态和业务逻辑的前提下单独适配。

## 实现边界

- 复用现有 `StageClassEntryControl`，不创建桌面/移动两套组件。
- 挑战开放逻辑继续由进度策略服务负责，UI只过滤和展示结果，不复制主线里程碑常量。
- 不修改挑战策划表、职业策划表、存档 schema、试炼策略或战斗入口。
- 不运行或覆盖任何 `public/` 生成内容。

## 测试与验收

- 第一章完成前：只显示主线按钮，不存在挑战面板入口。
- 第一章完成后：挑战按钮出现，挑战列表只包含挑战1~3。
- 第二章完成后：列表累计包含挑战1~6；第三章后累计包含挑战1~9。
- 无开放挑战但 `defaultMode='challenge'` 时仍渲染主线地图。
- CSS 回归测试确认桌面/平板主线入口底部为地图内12px，移动端现有映射不在本轮改变。
- 运行 StageSelect 聚焦测试、全量测试、lint、生产构建，并在已打开的演示页面进行桌面人工检查。
